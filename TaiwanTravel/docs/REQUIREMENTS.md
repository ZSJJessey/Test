# 台湾旅游攻略互动 Demo - PRD & 需求说明文档

## 1. 项目名称

台湾 6 天 5 晚自由行互动攻略 Demo

## 2. 项目背景

当前小红书旅游攻略以长图文和多张图片拼接为主，用户虽然能看到完整内容，但对整体路线、节奏、预算和每日亮点的理解成本较高。
为了增强内容的可视化表达、互动感和“收藏即抄作业”的体验，需要将原有台湾旅游攻略升级为一个可交互的静态网页 Demo，并支持通过 GitHub Pages 直接访问展示。

## 3. 项目目标

- 把原本线性攻略内容升级为可交互单页网页
- 提升用户对路线、时间线、准备事项和亮点内容的理解效率
- 增强“停留感、收藏感、分享感”
- 支持 GitHub Pages 一键部署和直接访问
- 方便后续继续扩展其他城市或其他旅行攻略模板

## 4. 目标用户

### 核心用户

- 大马留学生
- 第一次去台湾自由行的人
- 想节省做攻略时间的人
- 喜欢看可视化攻略、路线图和总结卡片的用户

### 次级用户

- 想参考路线但会二次调整的人
- 喜欢收集高质量旅行模板的用户
- 想把攻略做成作品集/项目展示的人

## 5. 用户痛点

- 长图文信息密度高，但不容易快速浏览
- 行程是按天写的，但整体路线感不强
- 准备事项、交通提示、现金/支付提醒容易被忽略
- 图片只是“展示”，没有和攻略信息形成联动
- 用户看完后仍然不知道“该先看哪部分”

## 6. 解决方案

将内容改造成单页互动式旅游攻略，页面结构围绕：

- 总览
- 时间线
- 每日详情
- 交通与准备
- 美食推荐
- 避坑总结

进行组织。

通过点击 Day 卡片、切换准备事项标签、展开推荐内容等方式，让用户按自己的阅读习惯获取信息。

## 7. 产品形态

- 类型：静态单页网站
- 技术方案：HTML + CSS + JavaScript
- 部署方式：GitHub Pages
- 开发工具：Cursor
- 适配端：优先移动端，同时兼容桌面端

## 8. 信息架构

1. 首屏 Hero
2. 行程总览
3. 准备事项
4. 每日时间线
5. 美食推荐
6. 避坑建议
7. 总结与 CTA

## 9. 页面模块清单

### 9.1 Hero 首屏

内容（与 `index.html` 保持一致）：

- 标题：台湾 6 天 5 晚自由行互动攻略
- 副标题：第一次去台湾，照着玩基本不踩雷 — 路线、准备与每日节奏一眼看清。
- 标签：自由行 / 路线清晰 / 可抄作业
- 主要按钮：查看路线、展开全部行程

交互：

- 点击按钮滚动到时间线区域
- 页面初次加载有轻微淡入动效 — **本期不做 / 已延期**（静态站保持首屏稳定，避免额外动效影响 LCP）

### 9.2 总览信息卡

内容：

- 总天数
- 覆盖城市
- 推荐人群
- 关键词：夜景 / 老街 / 湖景 / 日出 / 夜市 / 山景
- 预算感参考（人均区间；不含机票）
- **已实现**：简易预算计算器（右下角「算算预算」，纯前端）

交互：

- 卡片 hover 有高亮
- 移动端支持横向滑动查看
- 计算器：展开面板、输入变更实时更新人均与合计

### 9.3 出发前准备

内容：

- 电话卡
- 悠游卡
- 现金准备
- Google Maps
- 入境单
- 提前订票提醒

交互：

- 使用标签切换：证件 / 交通 / 支付 / 必备工具
- 点击标签后切换对应内容卡片

### 9.4 行程时间线

内容：

- Day1 台北：桃园机场 + 西门町 + 象山
- Day2 淡水
- Day3 故宫 + 九份
- Day4 日月潭
- Day5 阿里山
- Day6 日出 + 返程 + 桃园购物

交互：

- 点击某一天展开详情
- 当前选中日期高亮
- 展开内容包括：上午 / 下午 / 晚上 / 交通 / 美食 / tips

### 9.5 路线地图感模块

内容：

- 用简化路径示意展示地点顺序
- 台北 → 淡水 → 九份 → 日月潭 → 阿里山 → 桃园
- **已实现**：Google 地图 iframe 嵌入 + 驾车多途经外链（方案 1，无 Maps JS API）
- **地图定位说明**：节点文案仍为「九份」，Google 检索/途经关键词为 **台北车站**（住宿枢纽；与误标九份老街区分）

交互：

- 点击路线节点，同步切换到对应 Day 内容
- 当前节点有动效高亮
- 点击节点同步更新「在 Google Maps 中打开：当前站点」链接

### 9.6 美食推荐模块

内容：

- 品都串烧
- 天天利卤肉饭
- 葱爆山猪肉
- 民族烤肉/鸡翅包饭等

交互：

- 卡片翻转或展开
- 显示推荐理由、场景和个人主观评分

### 9.7 避坑与建议

内容：

- 现金一定要准备
- 公车票提前订
- 跨城市时间留宽松
- 回程班次要提前查
- 有些热门店不一定最好吃

交互：

- 红黄绿标签区分：必做 / 可选 / 避坑

### 9.8 总结区

内容：

- 最推荐 3 件事
- 适合什么人照着玩
- 你的真实体验声明
- 引导跳转到小红书或 GitHub 项目

交互：

- CTA 按钮：查看完整攻略 / 复制路线 / 访问 GitHub

## 10. 图片设计需求

### 图片类型分层

1. 氛围封面图

- 用于 Hero 首屏
- 建议选最有记忆点的一张：象山夜景、日月潭湖景、阿里山日出三选一

2. 每日代表图

- 每个 Day 一张图
- 作用是快速建立“这一天是什么感觉”

3. 信息型图卡

- 把电话卡、悠游卡、现金、订票提醒等内容做成小图标卡片
- 形成内容节奏，减少纯文字阅读疲劳
- **本期不做 / 已延期**：准备区已用 Tabs + 文字信息卡实现，独立「信息型配图卡片」留待下一迭代

### 图片使用原则

- 图片不只是展示，而要服务信息理解
- 每张图都要有标题或说明
- 首屏图突出情绪，日程图突出地点识别，卡片图突出信息效率

## 11. 可视化需求

### 核心可视化形式

- 时间线：展示 Day1-Day6
- 路线图：展示地点顺序
- 标签卡：展示准备事项和避坑建议
- 信息卡：展示预算感、交通方式、推荐指数

### 不建议第一版做的内容

- 不接真实地图 API
- 不做复杂登录/后台
- 不做多页面路由
- 不做数据库或后端服务

## 12. 交互需求

- 支持点击 Day 卡片展开详情
- 支持准备事项标签切换
- 支持路线节点与 Day 内容联动
- 支持一键展开全部内容
- 支持移动端滚动和桌面端 hover 状态
- 支持深浅色切换（可选）

## 13. UI 风格要求

- 风格关键词：清爽、旅行感、杂志感、轻量信息设计
- 避免廉价旅游模板风
- 配色建议：米白底 + 海岛蓝/山景绿点缀
- 字体建议：标题有轻微旅行海报感，正文清晰易读
- 整体优先级：信息清晰 > 装饰感

## 14. 技术需求

- 使用原生 HTML/CSS/JS
- 所有资源路径使用相对路径
- 入口文件为 index.html
- 不依赖后端
- 兼容 GitHub Pages 直接部署

## 15. GitHub Pages 部署要求

- 仓库根目录或发布目录中必须有 index.html
- CSS/JS/图片必须使用相对路径
- 推荐结构：
  - index.html
  - style.css
  - script.js
  - assets/

## 16. 验收标准

### 功能验收

- 页面可在浏览器正常打开
- Day 切换正常
- 标签切换正常
- 页面无白屏
- GitHub Pages 地址可直接访问

### 体验验收

- 用户 5 秒内看懂这是什么页面
- 用户 15 秒内能找到路线和准备事项
- 用户能快速知道“这篇适不适合我”
